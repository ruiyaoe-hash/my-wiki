# v1.3.0 立项方案 — 壳：结构化命令层 + 网页控制台

> 状态：已确认方向（2026-07-19，库主拍板），待施工
> 前置：v1.2.2 内核完成（协议引擎、状态、记忆、事件总线全部可用）

## 一、背景与问题

内核能干活，但入口是"懂系统的人"的入口：

- 命令散落：`agent-runtime`、`scripts/ingest.py`、`scripts/rebuild_sidecars.py`、
  `scripts/validate_sidecars.py`、`python -m unittest`，各有参数，不查文档不会用
- 状态埋点深：系统健康状况要自己去翻 `state/*.json`、`reports/`、`event_bus/history/`
- 结论：内核之上缺一层"壳"，真人用户（非技术背景）无法独立操作系统

## 二、目标

库主（不读源码、不记命令、不靠 AI 代劳）能独立完成完整工作流：
入库文章 → 检查健康 → 收尾归档，并随时看到整盘状态。

## 三、范围（两步走）

### Step 1：结构化命令层（地基）

`agent-runtime` 扩展为子命令结构：

```
agent-runtime check      # 跑 protocol/check.json，输出健康摘要
agent-runtime ingest <file> [--title ...] [--domain ...] [--tags ...]
agent-runtime wrapup     # 收尾归档
agent-runtime status     # 读 state/ + reports/，输出系统状态总览
```

- 薄分发层：只做参数解析，转发给现有 Executor/Planner，不复制内核逻辑
- 每个子命令支持 `--help`、`--json`（机器可读输出，供网页控制台调用）
- 旧入口（`scripts/*.py`）保留可用，文档引导到新命令

### Step 2：网页控制台（用户界面）

本地网页，Python stdlib `http.server`，只监听 `127.0.0.1`，零第三方依赖。

- **按钮区**：从 `protocol/*.json` 声明式生成，一个协议一个按钮；
  点按钮 = 替你敲 Step 1 的命令。加新协议按钮自动出现，不改网页代码
- **表单**：ingest 等需要参数的动作，网页填表单（文件路径、标题、标签），
  不用记命令行参数
- **健康总览**：知识页数量、过期页、断链数、当前任务、最近事件流水，
  一屏看完
- **写操作保护**：按钮会真实改库，必须有
  1. 二次确认（危险动作弹确认）
  2. 操作日志（谁点了什么、结果如何，追加落盘）
  3. 走 state_manager 既有锁机制，防止与命令行任务打架

## 四、明确不做

- 不嵌聊天对话：对话入口是 Kimi CLI（AI 代劳复杂决策），网页不管聊天
- 不引入第三方依赖（遵守 CONTRIBUTING 零依赖原则）
- 不做多用户、不做远程访问，只服务本机单人
- 不改内核：Executor/Planner/State/Memory 逻辑零改动，壳只调用

## 五、架构关系

```
网页控制台（Step 2）──┐
                      ├──> 结构化命令层（Step 1）──> Executor / Planner / State / Memory
Kimi CLI（AI 代劳）──┘
```

两条路走同一个引擎，状态永远一致；网页和命令行不会出现两套逻辑打架。

## 六、验收标准（每条需运行证据）

1. 库主不读源码，在网页上点按钮完成 check → ingest → wrapup 全流程，
   并在健康总览看到结果
2. CLI 子命令与网页按钮走同一引擎：网页点 check 后，`agent-runtime status`
   能看到这次执行记录
3. `python -m unittest discover tests` 全绿，新增测试覆盖命令层参数解析
   与网页后端接口
4. 零第三方依赖（`pip install -e .` 不装任何新包）

## 七、风险与对策

| 风险 | 对策 |
|------|------|
| 按钮误触发改坏库 | 二次确认 + 操作日志 + state 锁 |
| 网页与内核逻辑脱节 | 按钮从协议 JSON 生成，网页零业务逻辑 |
| 端口占用/多开 | 启动时检测，占用则换端口并提示 |
| Windows 编码老问题 | 沿用 executor 的 stdout reconfigure 方案 |
