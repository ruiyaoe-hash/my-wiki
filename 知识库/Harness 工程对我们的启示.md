# Harness 工程对我们的启示

## 一句话

**研究 Harness 工程，就是在研究怎么让我们手里的 AI 工具变得更能干、更自主、更靠谱。** 这篇文章不是纯学术——它直接关系到我们每天怎么用 Claude Code、Codex、飞书这些工具。

## 二八定律：七个核心投入产出比最高的点

80% 的价值来自这 20% 的理解：

### 1. 文件系统 > 上下文窗口（最重要！）

**Harness 三大设计模式中最核心的一条：不要把一切都塞进上下文。**

我们现在做的事情就是对的方向——用文件（知识库/、源/、hot.md、index.md）做持久化记忆，而不是每次对话都从头回忆。DGM、Meta-Harness 这些前沿系统用的也是同一套逻辑：把日志、实验记录、代码 diff 写进文件系统，用 grep/cat 查，而不是塞进 prompt。

**直接行动**：继续保持 Wiki 作为持久化记忆层的定位。每次会话的产物不要丢，写成知识页存着。

### 2. Harness 层比模型层更快迭代

翁荔的判断：RSI 会先发生在 Harness 层（脚手架），而不是模型权重层。这意味着：

- 我们不需要等下一个更强的模型出来才能提升 AI 工作流
- **改进我们的 AGENTS.md、协议文件、工具链配置，就是在做 Harness 工程**
- 我们的 `agents/` 目录（ingest-protocol.md、check-protocol.md、wrapup-protocol.md）就是在给 AI 配脚手架

**直接行动**：把优化协议文件当作正经的"Harness 迭代"来看。每次发现工作流卡壳，不是怪模型不够聪明，而是改协议。

### 3. 上下文工程是最直接能用的技术

ACE、MCE 这些听起来学术，但核心思想极其朴素：

- **ACE**：把上下文从"一段越来越长的提示词"变成"一本持续更新的操作手册"
- **MCE**：把"怎么管上下文"和"上下文里放什么"分开优化

**我们的对应**：
- `hot.md` = ACE 的"热缓存 playbook"
- `index.md` = ACE 的"结构化上下文索引"
- `agents/` 目录 = MCE 的"技能数据库"

**直接行动**：考虑把 agents/ 的触发条件再细化，让不同场景加载不同的"技能组合"——这本质就是 MCE 做的事情。

### 4. 自动化研究流水线的思路可以借鉴到文旅策划

AI Scientist、Autodata 这些系统虽然是学术研究方向的，但它们的流水线思路可以平移到文旅策划：

```
AI Scientist 流水线：    想法生成 → 实验 → 论文写作 → 评审
文旅策划的对应：        创意生成 → 数据验证 → 方案撰写 → 评审
```

**直接行动**：可以用 TPIS Pro 的 L0-L9 十阶段方法论作为 Harness 工作流的 backbone，让 AI 在其中自动跑某些环节。

### 5. 失败记录比成功记录更有价值（但我们都没存）

翁荔指出的第三个挑战：**负面结果被系统性忽视**。AI 不擅长判断"什么时候该放弃"，因为训练数据里成功远多于失败。

这对我们的启示非常直接：**建一个"踩坑日志"**。每次 AI 做错一件事或走了一条死路，不是删掉重来，而是记下来这个模式、原因和解决方案。这就像是给 AI 铺了一条"别走这条路"的路标。

**直接行动**：考虑在 Wiki 里开一个"踩坑集"页面，或给 log.md 增加一个"失败模式"字段。

### 6. 多样性 > 单一最优解

进化搜索的经验教训：如果只追高分，种群会坍缩到同一个模式。

对我们来说：**不要只用一种工具链、一种工作流**。保留备选路径，偶尔用用不同的方法——这本身就是防止"多样性坍缩"。

**直接行动**：能力地图里已经标注了备选工具链，继续保持"首选+备选"的双轨制。

### 7. 人的位置：往上走，不是被替代

翁荔说的第七个挑战——人类的角色——原话是："人类应当在整个技术栈中向上移动，而不是被排除在循环之外。"

对我们来说：**Codex/Claude Code 不是替代我们的工作，而是把我们推到更高层——从执行者变成审核者和策略制定者。**

- 以前：我们一行行写代码/写文档/查资料
- 现在：我们定义目标、审核输出、调整协议、维护知识库

**直接行动**：保持"先审后信"的习惯。AI 产出的每一页知识、每一段分析，都要过人的眼睛。

## 给文旅策划的具体建议

TPIS Pro 方法论 (L0-L9) 本质上就是一个 Harness——它定义了 AI 在文旅策划中"怎么思考、按什么流程、产出什么"。

从 Harness 工程的视角看，TPIS Pro 可以这样优化：

1. **把 L0-L9 每一级做成结构化上下文模板**（类似 ACE 的 playbook 条目）
2. **给每一级配评估标准**（没有好评估就没有好 Harness）
3. **建一个"项目踩坑集"**——记录哪些定位策略在什么条件下失败了
4. **用文件系统管理长项目**——不把一个完整策划案塞进上下文，而是分文件、分阶段管理
