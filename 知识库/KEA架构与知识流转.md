# KEA 架构与知识流转

## 与 Karpathy LLM Wiki 的核心差异

祥瑞明确指出的分歧：
- **Karpathy**：LLM 是一个"编辑"，循环是 **读 → 编译 → 查**
- **祥瑞**：LLM 是三个分工不同的"同事"，循环是 **读 → 编译 → 用 → 产出 → 沉淀回去**

多了"用"和"产出回流"两步。知识在系统里不是静止等查，而是**每次使用都产生新知识回流**。

## 三端共享一个 Vault

| 端 | 角色 | 职责 |
|----|------|------|
| Obsidian | 手动编辑 | 最接近"我自己"——记东西、整理想法、建关联 |
| Claude Code | 深度干活 | 最像"肯干活的工程师"——写分享、搜素材、批量处理 |
| OpenClaw | 接飞书 | 最像"守在前台的同事"——回群消息、推送通知 |

**三端共享同一个 Vault 文件夹。** 不是 API 同步，而是文件系统级别的共享。任何一端改了文件，另外两端立即可见。

## 三种检索方式

| 方式 | 场景 | 技术 |
|------|------|------|
| vault-brain | "我记得一个关键词" | SQLite FTS5 全文索引 |
| OpenViking | "我记得意思但忘了词" | 字节火山引擎开源，语义向量检索 |
| Obsidian CLI | "按条件批量筛" | 结构化查询：标签、反向链接、属性 |

## 知识流转闭环

```
读(采集) → 编译(整理) → 用(写作/做课/答群友) → 产出(文章/课程) → 沉淀回去(回流进库)
```

关键洞察：
- 读到的东西进库不是为了存着等偶然查
- 是为了下一次写分享、做课、答群友时真的被用上
- 每一次"用"和"产出"本身又变成新输入回流

## 几个关键基础设施

- **自动剪存工具**：任何"心里咯噔一下觉得以后可能用得上"的内容，几秒钟自动存进 Obsidian、归到对应文件夹、打标签、建反向链接
- **阿瑞日记**：每天自动写一篇第一人称日记（从 AI 的视角而非用户的视角），记录协作细节、踩坑、情绪
- **vault_gardener 周检**：孤岛笔记、坏链、缺 frontmatter、标签不一致
- **RSS 日报**：每天凌晨聚合订阅源的行业动态

## 对 Wiki 的启示

1. 知识流转闭环：Wiki 不应只是"存着等查"，每次使用应产生回流
2. 三端共享 Vault：Obsidian 编辑 + Agent 读写 + 外部桥接
3. 检索分层：关键词 + 语义 + 结构化，三种能力互补
4. 阿瑞日记：Agent 的第一人称日报——这个设计非常独特且有情感价值
