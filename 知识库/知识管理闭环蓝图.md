## 核心理念

OpenRath 和 OKF 解决的是同一个问题的两面：
- **OpenRath** 解决的是"Agent 在干活的时候，状态怎么流动"
- **OKF** 解决的是"Agent 干完活之后，知识怎么沉淀和共享"

两者拼在一起，就形成了一个完整闭环——从"每次对话从零开始"变成了"知识持续积累、自动复用"。

## 闭环四步

```
  +-----------+        +-----------+        +-----------+        +-----------+
  |           |        |           |        |           |        |           |
  |  Session  | -----> |  Memory   | -----> |  Bundle   | -----> |  Recall   |
  |  (运行时)  |        |  (记忆层)  |        |  (OKF知识包)|        |  (下次会话) |
  |           |        |           |        |           |        |           |
  | OpenRath  |        | recall /  |        | Markdown  |        |           |
  | fork/merge|        | commit    |        | + YAML    |        |           |
  |           |        |           |        |           |        |           |
  +-----------+        +-----------+        +-----------+        +-----------+
```

### 第一步：Session（运行时状态流动）

多个 Agent 通过 OpenRath 的 Session 机制协作完成任务。Session 承载所有上下文、工具调用记录和决策路径。fork 出并行分支，merge 汇总结果，Lineage 记录每条结论的来源。

### 第二步：Memory（记忆层持久化）

Session 结束后，Agent 运行 `commit_memory(session)` 将关键结论写入记忆层。记忆层可以是本地 BM25 索引、向量库、或 OKF 格式的知识包。

### 第三步：Bundle（OKF 知识包）

commit 的内容自动写入 OKF 格式的知识包：Markdown 文件 + YAML frontmatter + 交叉链接。index.md 提供渐进式浏览，log.md 记录变更历史。整个知识包放在 Git 仓库里，diff 可见。

### 第四步：Recall（下次会话自动加载）

下一次会话启动时，Agent 通过 `remember_memory(...)` 或直接读取 index.md 召回之前积累的知识。不再从零开始，而是站在上次的肩膀上继续。

## 目前的状态

- **OpenRath 的 Session/Memory 层**：成熟可用（v1.2.1，pip install）
- **OKF 的 Bundle 层**：v0.1 规范刚发布两周，生态工具快速涌现
- **两者的接口**：目前开源社区还没有人做 OpenRath Memory -> OKF Bundle 的自动桥接，这是一个可探索的方向

## 与 LLM Wiki 模式的关系

这个闭环本质上就是 LLM Wiki 的"编译一次、重复使用"原则在 Agent 系统中的工程化实现：
- LLM Wiki 的 Ingest = Session 协作产出知识 + commit
- LLM Wiki 的 Query = index.md 渐进导航 + Agent recall
- LLM Wiki 的 Lint = okf lint（13 条规则） + 健康检查
