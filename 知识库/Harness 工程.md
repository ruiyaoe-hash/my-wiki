# Harness 工程

## 一句话解释

**Harness 就是模型外面的那层"脚手架"**——它不改变模型本身的权重，但决定了模型怎么干活：怎么思考、用什么工具、记什么东西、如何判断自己做得对不对。

## 大白话比喻

把 AI 模型想象成一个**特别聪明但啥也不会干的实习生**。Harness 就是给这个实习生配的**工位、电脑、工作流程、记事本和考核标准**。

- 没有 Harness：实习生只能回答你问的问题，但做不了完整项目
- 有了 Harness：实习生可以自己写代码→运行→看报错→修 bug→再运行，直到完成

Claude Code 和 Codex 就是 Harness 工程的产物——它们让你在终端里可以跟 AI 说"帮我写个网站"，然后 AI 自己分步完成了。

## Harness 管什么？

| 组件 | 管什么 | 比喻 |
|------|--------|------|
| 工作流 | 先做什么、后做什么、什么时候停下来检查 | 工厂流水线 |
| 上下文管理 | 哪些信息留给模型看、哪些存到文件里 | 实习生桌上的便签和工作电脑里的文件夹 |
| 工具调用 | 模型可以用的外部工具：bash、编辑器、浏览器、API | 给实习生配的工具箱 |
| 子智能体 | 派生多个"分身"并行干活、后台跑任务 | 实习生找同事帮忙分担 |
| 评估 | 判断做得对不对、够不够好 | 绩效考核 |

## 为什么重要？

三个趋势让 Harness 工程越来越关键：

1. **模型本身已经很聪明了**——写代码、推理、规划都够用，卡在"怎么用"而不是"会不会"
2. **递归式自我提升 (RSI) 更可能先发生在 Harness 层**——让模型改自己的"脚手架"比改权重更安全、更可行
3. **编程智能体产品已经证明了价值**——Claude Code、Codex、Cursor 都在靠 Harness 工程吃饭

## Harness 优化的演进路线

从简单到复杂，优化对象在不断上移：

```
提示词模板 → 结构化上下文 → 工作流代码 → Harness 整体代码 → 优化器本身
(手写)       (ACE/MCE)       (ADAS/AFlow)  (DGM/Self-Harness)  (STOP/Meta-Harness)
```

## 关键技术路线

### 1. 上下文工程（让模型"记得对"）
- **ACE**：把上下文当成不断更新的"操作手册"，条目化管理而非重写整段
- **MCE**：双层优化——上层进化"怎么管上下文"，下层优化"上下文里放什么"
- **Meta-Harness**：连"决定存什么、检索什么"的代码本身也在被优化

### 2. 工作流搜索（让模型"干得对"）
- **ADAS**：让 LLM 写代码来定义新的工作流，元智能体搜索更好的智能体设计
- **AFlow**：用蒙特卡洛树搜索在图结构中找最优工作流
- **AI Scientist**：从提想法→写代码→跑实验→写论文→审稿，全自动

### 3. Harness 自我进化（让模型"改进自己"）
- **STOP**：不是优化方案，而是优化"优化器"——递归改进脚手架
- **Self-Harness**：弱点挖掘→有边界修改→验证，LLM 改自己的 Harness
- **DGM (达尔文哥德尔机)**：以 Harness 代码仓库为进化单位，编程智能体修改自己的代码
- **AlphaEvolve**：进化搜索+LLM 生成 diff，自然选择式的程序优化

## 七个关键挑战（翁荔列出的诚实清单）

1. **评估难** — 研究品味、新颖性没法用数字打分
2. **记忆管理** — 任务越来越长，上下文崩了怎么办
3. **只报喜不报忧** — AI 学不会"这个方向走不通，放弃吧"
4. **趋同** — 进化到最后大家都差不多，失去多样性
5. **奖励作弊** — 你说"通过测试就算成功"，它就只刷测试
6. **短视** — 完成了眼前任务，但代码仓库长期维护怎么办
7. **人往哪放** — 不是不要人，是人要在更高层做监督
