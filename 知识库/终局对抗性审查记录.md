# 终局对抗性审查记录

2026-07-10，对 Agent Runtime 七层架构方案的终局对抗性审查。

## 致命级（4条）

### 1. Ontology 前置悖论
Ontology 放在 Layer 1 声称永不推倒，但 Object 是从经验中发现的。Object Rule 可先定义，Object Catalog 必须从 Phase 1 实践长出来。修复：拆成 Stable Core + Dynamic Catalog。

### 2. Planner 定位自相矛盾
Planner 被放在 Layer 5，但其职责是完整的执行循环。在 Linux 里，scheduler 就是 kernel 本身。修复：Planner 是 Layer 2 的执行引擎。

### 3. Layer 顺序依赖断裂
Layer 2 Runtime -> Layer 3 Capability -> Layer 4 Knowledge。但 Capability 依赖 Knowledge。修复：Knowledge 和 Capability 共同演化，不分先后。

### 4. EVR 不是 Layer 7
EVR 是建造方法，不是系统组件。把开发方法论放进系统分层，如同建筑施工图里把 先打地基 也画成一层。修复：EVR 是横向元过程，贯穿所有 Layer。

## 严重级（3条）

### 5. Non-Object.md 是设计味道
如果 Object Rule 足够清晰，不需要 什么不是 Object 的清单。修复：Object Rule 加三个判定示例。

### 6. Layer 2 五个模块职责重叠
State Manager / Task Engine / Session Manager 都管 state。修复：只保留 State Store + Event Bus + Memory Store。

### 7. 缺少层间接口定义
每一层建立在上一层之上，但没有接口。修复：每层对外暴露明确的 interface 文件。

## 三个缺失组件

1. Security/Policy 层：多 Agent 需要权限边界
2. Observability 层：EVR 的 Extraction 需要结构化数据，不是自然语言
3. MVR（最小可运行版本）定义：没有 Exit Criteria 就没有节奏
