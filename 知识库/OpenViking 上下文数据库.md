# OpenViking 上下文数据库

> 来源：[volcengine/OpenViking](https://github.com/volcengine/OpenViking) · 字节跳动火山引擎 · 2026 年 1 月开源 · AGPLv3 · Python + Rust
> 全称：OpenViking — The Context Database for AI Agents
> 官网：[openviking.ai](https://openviking.ai)

OpenViking 是一个专为 AI Agent 设计的开源上下文数据库。它的核心创新是用**文件系统范式**替代传统 RAG 的向量存储模型，统一管理 Agent 的记忆、资源和技能。

---

## 解决的五个痛点

| # | 痛点 | 描述 |
|---|------|------|
| 1 | 上下文碎片化 | 记忆在代码里、资源在向量库里、技能散落各处，无法统一管理 |
| 2 | 上下文需求暴涨 | Agent 长任务每步都产出新上下文，简单截断/压缩导致信息丢失 |
| 3 | 检索效果差 | 传统 RAG 扁平存储，缺乏全局视图，难以理解完整上下文 |
| 4 | 检索不可见 | 隐式检索链像黑箱，出错无法排查 |
| 5 | 记忆迭代弱 | 当前记忆只记录用户交互，缺少 Agent 自身的任务记忆 |

---

## 五个核心机制

### 1. 文件系统管理范式 → 解决碎片化

不把记忆编码成向量做相似度检索，而是**像管理本地文件一样**，在目录结构下组织 Agent 的记忆、资源、技能。目录替代了无结构的向量空间。

### 2. L0/L1/L2 分层加载 → 降低 Token 消耗

| 层级 | 加载策略 | 内容 |
|------|----------|------|
| **L0** | 始终加载 | 核心上下文（系统提示、身份） |
| **L1** | 条件加载 | 命中才读的详细指令、技能说明 |
| **L2** | 按需检索 | 语义搜索日志、历史记录 |

不是一次性全吞，而是按需激活。这一点和祥瑞 Vault 的 CLAUDE.md → agent/ 文件 → 按天日志的三层结构、以及我们 Wiki 的 thin AGENTS.md → agents/ → 知识库页面三层导航**完全同构**。

### 3. 目录递归检索 → 提升检索效果

结合目录定位和语义搜索，实现递归精准获取上下文——先通过目录结构缩小范围，再做语义匹配。比纯向量检索少了大量噪音。

### 4. 检索轨迹可视化 → 上下文可观测

支持目录检索轨迹的可视化，用户可以看到检索路径。当 Agent 行为异常时，可以回溯「它看到了什么上下文」，定位根因。

### 5. 自动会话管理 → 上下文自我迭代

自动压缩对话内容、资源引用和工具调用记录，提取长期记忆。让 Agent 越用越聪明，而不是越用上下文越长。

---

## 技术实现

- 本地部署：`pip install openviking`
- 支持五种 VLM 后端：火山引擎 Doubao / OpenAI / Codex / Kimi / GLM
- Rust CLI 可选组件
- 需要 VLM + Embedding 模型两个能力

---

## 与我们 Wiki 的关系

| 设计模式 | OpenViking | 我们的 Wiki |
|----------|-----------|------------|
| L0 始终加载 | 核心系统提示 | thin AGENTS.md + hot.md |
| L1 条件加载 | 命中触发 | agents/ 子文件（触发词激活） |
| L2 按需检索 | 语义搜索 | 知识库页面（Query 协议走 index） |
| 文件系统范式 | 目录结构管理上下文 | Obsidian Vault = Markdown 目录 |

我们的 AGENTS.md 拆分和它的 L0/L1/L2 三层加载是同一套设计模式。这意味着分层加载正在成为 Agent 上下文管理的基础范式——字节、Anthropic、祥瑞，多路人马独立收敛到同一个方向。

当前阶段（28 篇知识页）不需要部署 OpenViking 这样的重型基础设施。但它的设计蓝图可以作为 Wiki 规模增长后的语义检索方向参考。
