# 知识库的三次解耦：为什么你的 AI 助手今天记住你，明天就忘了

**每个人都遇到过这个场景。**

你花了一个小时跟 AI 聊一个复杂话题——比如分析你所在行业的竞争格局。你给它喂了五篇文章、三份报告、一段你的个人笔记。它回答得头头是道。你很满意，关掉窗口。

第二天你打开新对话，说"接着昨天的聊"。它说：**"请问您指的是什么话题？"**

忘了。全忘了。你昨天喂的那五篇文章、那段分析、那个结论——仿佛从来没存在过。

这不怪你。也不完全怪 AI。问题出在架构上。

而恰恰在最近两个月，有三件事接连发生，分别来自硅谷、谷歌和清华。它们背后指向的是同一个动作。这个动作，正在悄悄改变"AI 怎么记住你"这件事。

---

## 你的 AI 助手，本质上是个金鱼

在说这三件事之前，我们先搞清楚一个问题：为什么 AI 这么"健忘"？

答案很简单。今天绝大多数 AI 工具的工作方式是 **RAG**（检索增强生成）。流程是这样的：你扔给它一堆文件，它把文件切成碎片，存进一个数据库。你每次提问，它去数据库里翻出相关碎片，现拼一个答案。

听起来合理。但问题在于：**它只是"翻"，不是"学"**。

就像一个学生每次考试都现翻课本——翻到了能答对，翻不到就瞎蒙。考完了书一合，脑子里什么都没留下。下次考试，继续翻。

更麻烦的是，你问一个需要综合五份文件才能回答的问题，它得同时翻五个地方，拼拼凑凑。拼错了你也不知道——因为没有任何积累帮你交叉验证。

这就是 RAG 的天花板：**它能检索，但不能积累。**

---

## 第一次解耦：把知识从检索里拆出来

2026 年 4 月，前特斯拉 AI 负责人 Andrej Karpathy 发了一篇 Gist，很短，就两千字。标题叫《LLM Wiki》。

核心思想一句话就能说清楚：**别再每次都现翻了。让 AI 帮你建一个 Wiki。**

什么叫 Wiki？就是一个由 AI 自动维护的知识库。你每放进去一份资料，AI 不是"切碎存好等着翻"，而是**读完、理解、写成一页笔记，链接到已有的知识上**。

比如你放了一篇讲"OpenAI 新模型"的文章。AI 不会把它切成三十块塞进数据库。它会：
- 创建一个"OpenAI"的实体页，写上这家公司的最新动态
- 更新"大模型发展史"这个概念页，把新模型的时间线补进去
- 检查有没有和旧文章矛盾的地方，标出来

你放了十篇文章之后，这个 Wiki 里就有几十页交叉链接的知识。你再问问题，AI 不是翻碎片，而是**翻自己的笔记**。

Karpathy 管这叫"编译一次，重复使用"。用大白话说：**知识不该每次现查，而该提前整理好。**

这就是第一次解耦——**把知识从检索里拆出来。**

---

## 第二次解耦：把格式从平台里拆出来

Karpathy 的想法很性感。但问题来了：你用什么格式存这些知识？

如果你用 ChatGPT 的格式，它就只能在 ChatGPT 里用。用 Claude 的格式，就只能在 Claude 里用。用某个创业公司的格式，万一它倒闭了，你的知识库就成了一堆打不开的乱码。

**格式不统一，知识就被锁在平台里了。**

2026 年 6 月，谷歌出手了。他们发布了一个叫 **OKF**（开放知识格式）的开源规范，v0.1 版。听起来很高大上，但其实就干了一件事：**规定知识库里的文件该怎么写。**

一个 OKF 知识库长什么样？就是一堆 Markdown 文件，每个文件顶部加几行结构化标签：

```
type: concept
title: "机器学习"
description: "让计算机从数据中学习的领域"
created: 2026-06-01
```

就这么简单。Markdown 是人类能读、任何编辑器都能打开的纯文本。那几行标签是给 AI 看的，告诉它"这是一个概念页，讲机器学习的，创建于六月一号"。

谷歌还发布了配套工具：一个 linter（帮你检查知识库有没有断链和矛盾）、一个 npm 包（方便程序员集成）、甚至一个合规校验工具（检查你的知识库是否符合 OKF 标准）。

Karpathy 点燃了"让 AI 建 Wiki"这个念头。谷歌把它写成了规范。从此，你在 Claude 里积累的知识，理论上可以无缝搬到 Codex、搬到任何支持 OKF 的工具里。

这就是第二次解耦——**把知识格式从平台里拆出来。**

---

## 第三次解耦：把状态从 Agent 脑子里拆出来

有了 OKF 规范，知识"怎么存"的问题解决了。但还有一个问题：知识"怎么产"？

假设你有五个 AI 助手在同时干活。一个分析数据，一个写报告，一个画图表，一个查资料，一个做校对。它们协作完之后，**这段协作过程本身，该被存成什么？**

聊天记录？太粗糙了——你看到的只是它们说了什么，看不到它们怎么想的、谁得出的结论、结论是怎么推导出来的。

这就引出了第三次解耦。

清华大学和中山大学的 Rath Team 开源了一个框架，叫 **OpenRath**。它的核心主张是：**Agent 不持有状态，Session 持有状态。**

什么意思？打个比方。Agent 是水管，Session 是水。水管只负责让水流过去，水本身不属于任何一根水管。Agent 干完活，产出就留在 Session 里，Agent 自己"脑子"是空的，随时可以被下一个任务复用。

更精妙的是，OpenRath 把 Session 设计成了类似 Git 的版本控制模型：

- 你可以 **fork** 一个 Session，同时探索两条不同的思路
- 你可以 **merge** 回来，把两个分支的结论合并
- 每个结论都有 **Lineage**（血统），记录它是从哪个分支、哪个 Agent、哪步推导来的

这意味着：**你不仅知道结论是什么，还知道结论是怎么来的。** 这在企业中特别重要——AI 给出的建议，如果出了问题，你得能追溯它的推理链。

这就是第三次解耦——**把状态从 Agent 身上拆出来。**

---

## 三件事串在一起，你就看懂了

现在我们把三件事放在一起看：

| 谁 | 做了什么 | 解耦了什么 |
|---|---|---|
| Karpathy | LLM Wiki 模式 | 知识 ≠ 检索 |
| 谷歌 | OKF 开放知识格式 | 格式 ≠ 平台 |
| 清华 | OpenRath 框架 | 状态 ≠ Agent |

这三件事不是竞争关系，而是**接力关系**。

Karpathy 提了想法："AI 应该帮我建 Wiki。"  
谷歌给了标准："Wiki 应该用这种格式写。"  
清华做了引擎："Wiki 应该由 Agent 的 Session 自动产出。"

它们拼在一起，就形成了一个完整的闭环：

**Agent 协作（OpenRath Session）→ 产出知识 → 写入 OKF 格式 → 下次会话自动加载**

从此以后，你的 AI 助手不再是一条金鱼。它能记住，能积累，能跟你一起成长。

---

## 还差一次解耦：我们发现的空白

不过，这个闭环还差最后一环。

OKF 管"知识怎么存"，OpenRath 管"知识怎么产"。但两者之间，**还没有一条自动的桥。**

什么意思呢？现在的情况是：Agent 用 OpenRath 跑完一个 Session 之后，产出了一些结论。但这些结论要变成 OKF 知识包，还得有人手动去整理、去写、去归档。

理想的情况应该是：**Session 一结束，Agent 自动把关键结论 commit 成 OKF 知识包。你下次开新会话，它自动把之前的积累加载进来。**

这个"自动桥接层"，目前开源社区还没有人做。

这是我们这群早期探索者在调研中发现的空白地带。它不算难——OpenRath 有 `commit_memory()` 接口，OKF 有完整的规范——就是还没人把它们接起来。

也许读到这篇文章的你，会成为做这件事的人。

## 这件事对你有什么用

说了这么多理论和框架，回到一个最朴素的问题：**我是一个普通人，这些跟我有什么关系？**

关系很大。因为它意味着你正在经历一场"AI 记忆方式的范式转移"。

过去两年，所有人都在卷"AI 怎么回答问题"。现在大家发现：**回答问题的前提是记住问题。** 记不住的 AI，再聪明也是每次从零开始。

所以接下来一两年，你会看到越来越多的产品不再标榜"我能回答什么问题"，而是标榜"**我能帮你积累什么**"。

对你个人来说，这意味着：

- **你不再需要每次跟 AI 聊天都从头解释一遍背景**。它知道你上周在做什么研究，知道你关注哪些话题，甚至知道你上次问到哪里卡住了。
- **你的知识不再散落在几十个聊天窗口里**。而是沉淀在一个你完全拥有、可以带走的 OKF 知识包里——纯文本文件，不依赖任何特定平台。
- **如果你是创业者或团队负责人**：OpenRath 的多 Agent 协作 + OKF 的知识沉淀，意味着你可以拥有一个**自主运转的 AI 知识团队**——Agent 负责干活和产出，知识自动积累和复用，不需要你花时间整理。

这不是科幻。Karpathy 自己已经在用这套方法管理他的研究。谷歌已经把 OKF 规范用到内部的 Claude Code 技能里。清华的 OpenRath 已经在 GitHub 上收获了关注。

**AI 正在从"能说会道"走向"能记能学"。**

而这场转变的核心，就是我们今天聊的这个动作——**解耦。**

把不该绑在一起的拆开。把该独立的独立。
