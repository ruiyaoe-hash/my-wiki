# SAG：基于SQL的大模型检索增强新范式——从GraphRAG痛点到多跳检索突破

> 来源: 得到大脑 · 创建: 2026-06-23

### **🔍 RAG技术基础与挑战（背景）**

**RAG核心原理**
- **定义**：检索增强生成（Retrieval-Augmented Generation），大模型回答前先从数据库检索相关材料，再组织回答。
- **典型场景**：企业知识库、智能客服、文档问答。
- **基础流程**：
  1. 资料转化为数据库（便于搜索比对）
  2. 提取问题关键信息，比对数据库找到语义最相关片段
  3. 将问题与相关片段输入大模型生成回答

**传统RAG痛点**
- **多跳检索局限**：当回答分布在多个文档时，向量搜索难以一次全量获取，需消耗更多token运行Agent。
- **GraphRAG瓶颈**：需构建实体-关系知识图谱，存在项目过重、图谱维护成本高、关系词提取不稳定（如"收购"与"并购"混淆）等问题。

### **💡 SAG创新方案：Event-Entity超边结构（核心方法）**

**核心思路**
- **提出者**：Jomy（赛博月刊搭档）
- **核心改进**：用整句**Event**保留完整语义，提取**Entity**作为索引入口，避免三元组拆分导致的语义丢失。

**三元组 vs Event-Entity超边对比**

| 维度 | 三元组拆法 | SAG做法（Event-Entity超边） |
| :--- | :--- | :--- |
| **处理方式** | 拆分为主谓宾三元组（如"A公司→收购→B公司"） | 保留整句Event，关键词Entity画线 |
| **原始文本示例** | "2024年3月，A公司以12亿美元收购了B公司，B公司CTO张三随后加入C项目担任技术负责人" | Event：完整保留原句<br>Entity：A公司、B公司、张三、C项目、12亿美元、2024.03 |
| **优势** | - | ✅ 语义完整保留<br>✅ 不依赖谓词抽取准确度<br>✅ 天然保留时间和因果关系 |
| **缺陷** | ❌ 同事件拆分为多片段<br>❌ 谓词提取不稳定（如"收购"vs"并购"）<br>❌ 时间、因果关系丢失 | - |
| **Recall@5性能** | **63.78%** | **80.04%**（结构选择提升16个百分点） |

### **📊 SAG性能验证：Benchmark结果（效果）**

**测试环境**
- **数据集**：HotpotQA、2WikiMultiHop、MuSiQue（多跳问答标准数据集）
- **配置**：Embedding = bge-large-en-v1.5，LLM = qwen3.6-flash

**关键指标对比（SAG vs HippoRAG 2）**

| 指标 | SAG | HippoRAG 2 | 提升幅度 |
| :--- | :--- | :--- | :--- |
| **平均Recall@2** | 79.30% | 68.14% | **+11.16%**（相对提升+16.4%） |
| **平均Recall@5** | 88.18% | 83.28% | **+4.9%** |
| **MuSiQue Recall@5** | 80.04% | 65.13% | **+14.91%** |
| **HotpotQA Recall@2** | 91.55% | 78.35% | **+13.2%** |

**结果说明**
- 在9项Recall@K指标中赢下8项，尤其在复杂组合式多跳问题（如MuSiQue）上优势更明显。
- 减少上下文切换次数，帮助Agent更快命中关键证据。

### **🔄 SAG多跳检索实现：SQL JOIN方案（技术细节）**

**多跳检索流程**
1. 获取初始Event集合
2. 通过SQL提取关联Entity
3. 基于Entity查询关联的其他Event（排除已有的）
4. 重复步骤2-3实现多跳扩展

**工程化能力**
- **数据规模**：已在5亿级数据上跑通
- **检索延迟**：线上环境秒级响应
- **应用案例**：Jomy的Zleap产品（企业本地化Agent数据底座）
