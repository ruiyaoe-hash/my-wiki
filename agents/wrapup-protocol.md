# 会话结束 (Wrapup) 协议

触发：用户说出"收尾"、"结束"、"wrapup"、"今天到这"等闭口信号。

## 七步流程

### 1. 创建会话归档页 → 归档/会话/
- 格式：type: session-archive，含 date、duration、domains
- 正文：会话主题、完成事项清单（按大类分组）、最终状态（wiki 文件数/结构变化）、待办
- 文件名：YYYY-MM-DD 会话归档.md

### 2. 更新 hot.md
- 当前状态摘要：知识库总页数、MOC 结构、标签数、源文件状态
- 本次成果：按大类分组的完成事项（一到两句话一组）
- 下次约定：P0/P1/P2 待办，具体到可执行的下一步
- 用户偏好：如有新增或改变，追加到末尾

### 3. 更新 log.md
- 顶部追加 `## [日期] archive | 会话归档` 条目
- 一句话概括本次会话核心成果
- 如果当天有多次记录，这一条作为汇总归档

### 4. 更新根 index.md（封面页）
- 更新「最近」段落——把本次会话的 3-4 条核心成果写入
- 原则：封面反映当前状态，不是操作日志。旧条目下沉，保持 5 条以内
- 「从这里开始」的域入口链接如有变动（如重命名），同步更新

### 5. 更新四层子目录 index
每层按对应原则更新：

| 子 index | 定位 | 需更新时 |
|---------|------|---------|
| 知识库/index.md | 目录——全部知识页清单 | 页面有新增/删除/重命名 |
| 源/摘要/index.md | 库存——源摘要清单 | 有新摘要入库 |
| 源/原文/index.md | 库存——原文清单 | 有原文入库/移出 |
| 归档/会话/index.md | 档案——会话记录索引 | 本次归档页写入后更新条目 |

### 6. 验证 MOC 链路
- 本次新建的知识页是否已挂到对应的域 MOC 或子 MOC 中？
- 本次新建的子 MOC 是否已被域 MOC 引用？
- 方法：打开 🏠 域 MOC → 顺着子 MOC 链路点到末页，确认无断头路

### 7. 剪枝 (Prune) — 借鉴 Claude Code Dream 系统
Claude Code 的 autoDream 服务四步：Orient→Gather→Consolidate→**Prune**。我们缺的就是最后这一步——只管加不管减。剪枝做三件事：

- **待办降级**：检查 hot.md 的「下次会话约定」。超过 3 次会话仍无实质进展的 P0 条目，降级为 P1/P2 或直接删除。口头上的 P0 不算 P0。
- **Stale 页标记**：扫描知识库，找到距 `updated` 超过 90 天且正文不足 200 字的页面。将其 `status` 更新为 `stale`（不改正文）。
- **Protocol-gap 汇总**：检查 log.md 中本月所有标记了 `[protocol-gap]` 的条目，汇总为一条写入 hot.md 的「下次约定」，确保下次会话优先处理协议改进。

剪枝完成后，在 log.md 顶部追加一行摘要：`剪枝：降级 X 条待办 / 标记 Y 个 stale 页面 / 发现 Z 个 protocol-gap`

## 纪律

1. 收尾信号一旦发出，七步不可跳过
2. 操作完成后必须同步更新所涉文件的 frontmatter `updated` 字段为当前日期
3. 归档页本身也要有 `updated` 字段
4. 全部完成后在回复中列出每项更新摘要（哪一步、写了什么、影响哪些文件）

## 与 ingest-protocol.md 的关系

- ingest 是「吃进去」的流程——单篇文章/项目入库
- wrapup 是「关门」的流程——整场会话收摊
- 两者各自独立触发，不互相调用
- 两者的 frontmatter 铁律（updated/source/回链）一致
