# Wiki 知识生命周期引擎

> 触发词：用户说「wiki 进化」「引擎启动」「知识升级」
> 架构：4 研究 Agent + 2 挑刺 Agent + 1 合成 Agent
> 输出：P0/P1/P2 三级 wiki 改进建议，直接可执行

---

## 触发条件

用户说出以下任一触发词时加载本协议：
- 「wiki 进化」
- 「引擎启动」
- 「知识升级」

## 执行流程

### 第一阶段：并行研究（4 个 Agent 同时工作）

每个研究 Agent 独立扫描自己的领域，不互相通信。

| Agent | 研究域 | 搜索方向 | 对标 wiki |
|-------|--------|---------|-----------|
| 研究-A | AI 工程 | Agent 架构、记忆系统、认知框架、多智能体系统的最新论文和实践 | 🏠 AI工程 |
| 研究-B | 元知识 | 知识管理理论、Wiki 方法论、知识表示、信息架构的学术和实践前沿 | 🏠 元知识 |
| 研究-C | 工程实践 | 开源项目、工具链、社区最佳实践、可复现的技术方案 | 跨域技术评估 |
| 研究-D | PKM | Obsidian 生态、PKM 工具演进、社区工作流模式、个人知识管理趋势 | 跨域工具评估 |

搜索要求：
- 每个 Agent 搜索至少 3 个来源（学术论文、开源仓库、技术博客、社区讨论各至少覆盖一次）
- 优先找 2025-2026 年的内容
- 每条发现必须有可追溯的 URL
- 输出格式：`[发现标题] - 一句话总结 - 来源 URL`

### 第二阶段：对抗审查（2 个 Agent 独立工作）

挑刺 A 和挑刺 B **共享四个研究 Agent 的原始输出**，但**互相看不到对方的点评**。

| Agent | 挑刺角度 | 具体检查 |
|-------|---------|---------|
| 挑刺 A | 逻辑漏洞 | 研究结论之间是否自相矛盾？是否有被忽略的反面证据？假设是否可被推翻？ |
| 挑刺 B | 证据缺口 | 每条结论的支撑证据是否充分？有无遗漏的关键论文/项目/作者？时效性是否可靠？ |

审查规则：
- 每条质疑必须指向具体的研究结论，不能笼统否定
- 质疑本身就是有价值的输出——即使原结论成立，质疑点也可能是 wiki 的边界条件或风险提示

### 第三阶段：统合分析（1 个合成 Agent）

合成 Agent 接收：四个研究原始输出 + 两个挑刺报告。

对照 wiki 现状进行分析：
1. **覆盖缺口**：哪些研究结论在 wiki 里完全没有对应页面？
2. **深度不足**：哪些 wiki 页面只提了概念，缺了最新的实践参考？
3. **过时风险**：哪些 wiki 页面的内容已被新研究/新工具超越或推翻？
4. **结构优化**：wiki 的组织方式（平铺、MOC 导航、标签索引）有没有更优解？

输出 P0/P1/P2 三级改进建议，每级不超过 5 条。

### 第四阶段：写入

将改进建议写入 `知识库/知识管理闭环蓝图.md` 的「待办」段落，同时在 log.md 追加记录。

## 设计原则

1. **独立性保证**：研究层之间不通信（防止群体思维），挑刺层之间不通信（保证审查独立性）
2. **可追溯性**：每条发现必须有 URL，每条建议必须能回推到具体研究结论
3. **可执行性**：输出不是「建议考虑 XXX」而是「在 XXX 页面增加 YYY 段落，参考 ZZZ」
4. **对抗不为否定**：挑刺的目标是让结论更牢靠，不是推翻一切——质疑本身就是有价值的知识资产
